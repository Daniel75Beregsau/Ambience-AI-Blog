rag_service/src/api/routes.py 
async def generate_with_fallback(query: str) -> Response:
    try:
        # Try local model first
        result = await local_model.generate(query, timeout=8)
        return result
    except TimeoutError:
        # Fall back to cloud on timeout
        try:
            result = await cloud_model.generate(query, timeout=15)
            return result
        except Exception:
            return low_confidence_refusal()
    except Exception as e:
        logger.error(f"Generation failed: {e}")
        return user_friendly_error_response()

Weeks 17–18

RAG Integration and Cloud Routing

With the frontend and backend already stable and the infrastructure pivot resolved, this period was about connecting the RAG service to the rest of the system and finalising the routing strategy between local and cloud models. This was the heart of the February push — turning three separate services into one working product.

RAG Service Connected to Backend

The RAG service was now fully integrated with the main backend. When a GP submits a clinical question, the backend forwards it to the RAG service, which retrieves relevant guideline evidence, generates a grounded response, and streams it back through the consultation interface.

  • Document ingestion: Complete pipeline from PDF upload to indexed vector chunks
  • Retrieval: Hybrid vector similarity + keyword search with BM25 re-ranking
  • Generation: Med42 generates responses grounded in retrieved evidence
  • Citation mapping: Exact substring matching traces claims back to source guidelines

Routing Strategy Implementation

The router directs queries to local or cloud models based on available resources:

rag_service/src/generation/router.py 
class ModelRouter:
    async def select_provider(self, query: str) -> str:
        # Check if local Ollama model is available
        local_available = await check_ollama_health()

        if local_available:
            return "local"   # Default: local Med42 8B
        else:
            return "cloud"   # Fallback: RunPod 70B

Retrieval Quality Tuning

With the full pipeline running end-to-end for the first time, we could now measure and improve retrieval quality against real queries:

  • Hybrid search: Combined vector similarity (0.7 weight) + keyword overlap (0.3 weight)
  • Re-ranking: BM25 re-ranking on top-20 results from vector search
  • Citation precision: Exact substring matching from original guideline documents
  • Confidence scoring: Based on retrieval similarity and answer grounding quality

Weeks 15–16

Intel Server Loss and Cloud Pivot

Intel informed us that the Gaudi 2 server would need to be reclaimed as the contract was expiring. This forced a critical architectural decision: we could no longer run the Med42 70B parameter model on dedicated hardware, and we needed to find a sustainable alternative quickly — while the frontend and backend remained stable and working.

The Challenge

"We had built our model inference strategy around a powerful on-premise server, but we needed to pivot to something sustainable. The 70B model was too demanding for laptops, and unlimited cloud compute costs would not be viable for this project."

Evaluating Alternatives

We rapidly evaluated our options and converged on a hybrid approach:

  • Med42 8B (quantized): Lightweight enough to run locally via Ollama on standard hardware — discovered it ran well even on a MacBook Pro
  • Cloud endpoints: RunPod offered pay-per-use GPU pods for the full 70B model when needed
  • Hybrid routing: Use 8B locally as the default, route to cloud 70B only for complex queries or when local resources are unavailable

Local: Ollama with thewindmom/llama3-med42-8b (Q8_0 quantization, ~8 GB) | Cloud: RunPod pod with Med42 70B for demanding cases | Hardware: Standard MacBook Pro (16 GB RAM) sufficient for local path

backend/src/middleware/auth.py 
async def require_role(role: str):
    def dependency(
        token: str = Depends(oauth2_scheme),
        db: AsyncSession = Depends(get_db),
    ):
        payload = decode_jwt(token)
        user = await get_user_by_id(db, payload["sub"])
        if user.role != role:
            raise HTTPException(403, "Insufficient permissions")
        return user
    return dependency

Weeks 3–4

Requirements Baseline and Gaudi Planning

We wrote up what we learnt in the interviews into explicit functional and non-functional requirements, then prioritized them with MoSCoW. We also identified key risk areas and planned our initial deployment path.

MoSCoW Prioritization

Category Requirements Examples
MUST Evidence-grounded answers with citations All responses tied to source guidelines, specialist review capability, audit trail
MUST Specialist oversight and approval gates No direct GP→patient response, specialist review required, escalation paths
SHOULD Support for two clinical domains Neurology and Rheumatology (vs. single domain)
COULD Multiple medical document ingestion NICE, BSR, RCEM guidelines in single system

Key Risk Areas Identified

  • Model hallucinations: LLM generating plausible-sounding but incorrect medical information
  • Deployment uncertainty: Unknown performance of 70B model on Gaudi 2 architecture
  • Governance expectations: NHS and legal requirements around AI decision-making
  • Evidence base completeness: Ensuring all required guidelines are included

Initial Deployment Strategy

We planned the initial deployment path around Intel's Gaudi 2 server with the medical-trained Med42 LLM, accepting that we would need to validate feasibility early.

Weeks 1–2

Problem Framing and Specialist Interviews

We started by mapping the Advice and Guidance (A&G) pain points with specialist input from Rheumatology and Neurology. The shared message was clear: faster responses are useful only when evidence is visible and clinical accountability is preserved.

Research into A&G Workflow

General Practitioners regularly ask specialists repetitive questions about:

  • Monitoring thresholds for chronic conditions
  • Referral criteria and red-flag symptoms
  • Medication pathways and drug interactions
  • Test interpretation and imaging findings

Currently, finding correct NICE and BSR guidance in long documents is time-consuming, and generic LLM chat systems lack the transparency and audit trails expected in NHS workflows.

Specialist Interviews

We conducted interviews with:

  • Rheumatology consultant: Focused on DMARDs management and monitoring
  • Neurology consultant: Emphasis on multiple sclerosis and Parkinson's disease workflows
  • GP partners: Perspective on time constraints and practical needs
"We need faster guidance, yes, but not at the cost of accountability. Every recommendation must be traceable to evidence, and I need to review before any answer reaches the GP."

Project Scoping

From these conversations, we narrowed scope to neurology and rheumatology guidance so retrieval could be grounded in a realistic, manageable evidence base. We agreed that evidence needed to stay visible so the specialist could review the response before anything was officially submitted to a GP.

Initial Requirements Baseline

📋 Core Principles

1. Keep answers grounded in guideline evidence — Every claim traces to NICE/BSR documentation | 2. Preserve specialist oversight — No direct AI→GP responses, always reviewed | 3. Make responses easy to review — Evidence highlighted, confidence scores visible
